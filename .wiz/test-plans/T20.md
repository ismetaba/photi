# T20 Test Plan — Broadcast `photo-removed` on DELETE /photos/:id

## Acceptance Criteria
1. `FoyerEvent` union includes `{ eventId, type: 'photo-removed', photoId }`.
2. `DELETE /photos/:id` calls `foyerHub.broadcast({ type: 'photo-removed', ... })` only when the deleted photo was `isFeatured === true && status === 'ready'`. Otherwise no broadcast.
3. Web Foyer onmessage handler still refetches `/foyer-data` on every payload (covers `photo-removed`).
4. Existing happy paths (delete row + storage cleanup) untouched.

## Test Scenarios (Vitest)
- **backend photos-route**: seed `ready + isFeatured=true` photo via DB write, DELETE → spy `foyerHub.broadcast` called once with `{ type: 'photo-removed', eventId, photoId }`.
- **backend photos-route**: seed `processing` (default) or `isFeatured=false` photo, DELETE → spy not called.
- **web Foyer**: dispatch synthetic `photo-removed` MessageEvent on the mocked EventSource, assert `/foyer-data` refetched.
