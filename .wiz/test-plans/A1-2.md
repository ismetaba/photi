# A1-2: Foyer header must show the QR code

## Acceptance criteria

- [x] The Foyer page (`/e/:slug/foyer`) renders an `<img>` whose alt matches `/qr/i` in the header.
- [x] The QR is fetched from a **public** backend endpoint that does not require `x-user-id`.
- [x] Existing organizer-only `GET /events/:id/qr` continues to work unchanged.
- [x] Existing "F = tam ekran" hint remains visible (just relocated if needed).
- [x] Foyer rotation tests still pass (additive change).

## Decision: Backend public PNG endpoint (option 1)

We picked **option 1** — add a public `GET /events/:slug/qr.png` route — for two reasons:
1. The Foyer page is unauthenticated; piggy-backing on the existing slug-based `foyer-data` carve-out keeps the public surface coherent.
2. Reuses the existing `makeQrPng` service and the storage-free QR pipeline already used by the organizer route, without dragging a new browser dep into the bundle.

The new route lives **next to the existing `/events/:id/qr`** so both reads stay in `routes/events.ts`. The unauth carve-out is added in `middleware/identify.ts` similar to the existing `foyer-data` exception.

## Test scenarios

1. **Backend — public QR endpoint** (`apps/backend/src/__tests__/events-route.test.ts`)
   - GET `/events/:slug/qr.png` with NO `x-user-id` header → 200, `image/png`, PNG magic bytes.
   - GET `/events/unknown-slug/qr.png` → 404.
   - GET `/events/:id/qr` (the legacy organizer route) still requires auth and still works for the owner.

2. **Web — Foyer renders the QR `<img>`** (`apps/web/src/__tests__/participant-pages.test.tsx`)
   - Mock `fetch` so `/events/:slug/qr.png` resolves to a PNG blob.
   - Assert the rendered Foyer contains an `<img>` with an `alt` attribute matching `/qr/i`.
   - Assert the `src` is a `blob:` URL (created by `URL.createObjectURL`) or, if the implementation chose a direct URL, contains `/qr.png`.

3. **Web — "F = tam ekran" hint still rendered** (existing test) — the rotation tests must still pass.
