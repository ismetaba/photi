# T21 Test Plan — Block destructive photo ops on archived events

## Acceptance Criteria
1. `POST /events/:id/photos` returns 409 `{ code: 'event_archived' }` when the parent event has `status === 'archived'`. No Photo row, no Job row.
2. `PATCH /photos/:id` (`isFeatured` toggle) returns 409 `event_archived` for archived events. No foyer broadcast. Photo state unchanged.
3. `DELETE /photos/:id` returns 409 `event_archived` for archived events. Photo + storage objects remain.
4. Read endpoints (`GET /events/:id/photos`, `GET /me/photos`, `GET /events/:slug/foyer-data`) still respond normally for archived events.
5. Existing happy-path tests for `draft` / `live` events keep passing.
6. Events route untouched — `PATCH /events/:id`, `POST /events/:id/publish`, `POST /events/:id/archive` still work.

## Test Scenarios (Vitest)
- **archive then upload** → 409 `event_archived`; no rows in `photos` for the event; no `process-photo` jobs queued.
- **archive then PATCH /photos/:id** → 409; spy hub never sees a `photo-featured` event; row's `is_featured` unchanged.
- **archive then DELETE /photos/:id** → 409; storage entries still exist; row still in DB.
- **archive then GET /events/:id/photos** still 200 (read allowed).
