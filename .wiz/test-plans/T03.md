# T03 Test Plan — Backend DB Schema + Identify Middleware

## Acceptance Criteria
1. Drizzle schema covers `users`, `events`, `photos`, `participants`, `photi_transactions`, `jobs`.
2. Indexes: `photos(eventId, status)`, `participants(eventId, userId)`.
3. `db:migrate` script generates + applies SQLite migrations (or applies via `drizzle-orm/better-sqlite3/migrator` from a generated folder; for the in-memory test path we apply schema directly).
4. Fastify `identify` plugin reads `x-user-id`. Missing header → 400.
5. First request with a fresh `x-user-id` creates the user + writes a `signup_bonus` PhotiTransaction of +100 in a single SQLite transaction. Re-requests with the same id reuse the user (no extra bonus row).
6. `GET /me` returns `{ user, balance, transactions }` where `transactions` are the last 20 ordered newest-first.

## Test Scenarios (Vitest, backend package)
- **Schema unit tests**: schema object exports `users`, `events`, `photos`, `participants`, `photiTransactions`, `jobs`.
- **identify**:
  - Request without `x-user-id` → 400 with code `missing_user`.
  - Request with malformed user id → 400 with code `invalid_user_id`.
  - First request with valid uuid creates user with `photiBalance` 100; one `signup_bonus` row exists with amount +100.
  - Second request with same id reuses the user; only one signup_bonus row.
- **GET /me**:
  - Returns `user.id`, `balance: 100`, exactly one transaction with type `signup_bonus`.
  - After we insert extra transactions for that user, returns last 20 newest-first.
- **Concurrency invariant**: signup bonus runs in a transaction → if we manually attempt to insert two signup bonuses for the same user, only one succeeds (unique constraint or guarded insert).
