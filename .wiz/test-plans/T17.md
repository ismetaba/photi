# T17 Test Plan — Friendly 413 + per-part oversize handling

## Acceptance Criteria
1. Fastify `setErrorHandler` maps:
   - `err.code === 'FST_REQ_FILE_TOO_LARGE'` and `err.code === 'FST_FILES_LIMIT'` → 413 `{ code: 'file_too_large', limitBytes }`.
   - Other `FST_*` codes → 400 `{ code }`.
   - Otherwise → default Fastify behaviour.
2. `POST /events/:id/photos` collects oversize parts in `rejectedTooLarge[]` (not aborting the batch), returning 201 with `{ created, rejected, rejectedTooLarge }`. If everything is rejected (any reason), still 415 `no_supported_images` with both arrays.
3. `POST /participants/:id/selfie` returns 413 immediately when a part is too large; no DB or storage side-effects.
4. The 25 MB / 200-file caps remain unchanged.
5. Web Dropzone messages too-large filenames distinctly.

## Test Scenarios (Vitest)
- **photos route oversize**: build a buildTestServer with `multipartLimitBytes=64`. Upload one part with 200 bytes → returns 201 with `rejectedTooLarge` containing the filename, `created=[]`, `rejected=[]`. (Or 415 if no created.)
- **selfie route oversize**: same configuration → 413 `file_too_large`, participant fields untouched.
- **error handler unit**: post non-multipart body to a multipart route with a fake `FST_REQ_FILE_TOO_LARGE` error → 413.
