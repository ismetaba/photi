# T02 Test Plan — Shared Zod Schemas & TS Types

## Acceptance Criteria
1. `packages/shared/src` contains a file per entity: `user.ts`, `event.ts`, `photo.ts`, `participant.ts`, `photiTransaction.ts`, `job.ts`.
2. Each entity exports a Zod schema and an inferred TS type.
3. Status enums exist: `EventStatus`, `PhotoStatus`, `JobStatus`, `PhotiTxnType`.
4. API DTOs `CreateEventInput` and `FoyerData` are exported.
5. `packages/shared/src/index.ts` re-exports everything from each entity file plus DTOs.
6. Schemas accept good payloads and reject bad ones.

## Test Scenarios (Vitest)
- `UserSchema` parses a valid user; rejects negative `photiBalance`; rejects non-uuid `id`.
- `EventSchema` parses a valid event; rejects bad branding color; rejects unknown status.
- `PhotoSchema` parses with empty `faceVectors` and rejects when face vector length isn't 128; rejects unknown `status`.
- `ParticipantSchema` parses with `selfieKey`/`faceVector` null; rejects when faceVector length isn't 128.
- `PhotiTransactionSchema` parses for each `PhotiTxnType`; rejects unknown type.
- `JobSchema` parses queued/running/done/failed; rejects negative attempts.
- `CreateEventInput` parses minimal create payload, rejects missing title.
- `FoyerData` parses sample payload.
- Re-export sanity: `index.ts` exports all schemas and types.
