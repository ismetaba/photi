# T07 Test Plan — Job worker + face-api pipeline + billing

## Acceptance Criteria
1. `services/matching.ts` exports `cosineDistance(a, b)` and `MATCH_THRESHOLD = 0.5`.
2. `services/faceApi.ts` exports a `FaceEngine` interface with `detectAndEmbed(buffer): Promise<number[][]>`. The default implementation lazy-loads tfjs + @vladmandic/face-api (kept untested in unit tests).
3. `services/faceApi.ts` also exports a swappable factory + a `setFaceEngine(engine)` (or DI via `JobRunner({ faceEngine })`) so tests can inject a deterministic mock.
4. `jobs/processPhoto.ts`: given a photoId, loads the photo bytes, runs `detectAndEmbed`, persists `faceVectors`. For each event participant with a stored faceVector, computes cosine distance vs all photo vectors; collects participant userIds with `min(dist) < 0.5` not already in `matchedUserIds`. Debit organizer 1 Photi per new match in a single SQLite transaction (writes `distribution` PhotiTransactions). If organizer balance can't cover all new matches → set status `awaiting_credit`, do **not** write matches. Otherwise → status `ready`.
5. `jobs/matchSelfie.ts`: given a participantId, loads selfie embedding (already stored on Participant) and runs the inverse — for each `ready` photo with `faceVectors`, computes distance vs selfie vector and adds the user to `matchedUserIds` if matched. Bills organizer 1 Photi per new match. Identical insufficient-credit handling.
6. `jobs/queue.ts` exposes `runOnce(deps)` that picks one queued job, marks `running`, dispatches to type-specific runner, on success → `done`, on error → increment attempts, set `failed` on attempts >= 3.
7. Polling runner (`startQueueLoop`) sets a setInterval polling every 2000ms — wrapped behind a flag so tests don't start the loop.
8. `retry-awaiting` job: scans the user's events for `awaiting_credit` photos and re-enqueues `process-photo` for each. The user's events are passed via payload `{ userId }`.

## Test Scenarios (Vitest)
- **matching.cosineDistance**: identical vectors → 0; orthogonal → 1; threshold constant.
- **processPhoto - happy path**: setup event + 2 participants with deterministic vectors; mock face engine returns known vectors; run job; resulting photo has `matchedUserIds` covering matching participants; organizer balance dropped by N; PhotiTransactions of type `distribution` written; photo status `ready`.
- **processPhoto - insufficient credit**: organizer balance < new match count → status `awaiting_credit`; matchedUserIds NOT updated; no PhotiTransactions written.
- **processPhoto - retry idempotency**: running the same job twice does not double-bill (only new userIds count).
- **matchSelfie**: one photo with one face vector; participant has matching vector; running job adds user to matchedUserIds and debits 1 Photi. If photo's organizer has 0 balance → photo flips to awaiting_credit and no debit.
- **runOnce**: queued → running → done lifecycle; unknown job type marks failed; thrown error increments attempts; attempts ≥ 3 → failed.
- **retry-awaiting**: enqueues `process-photo` for the user's awaiting_credit photos.
