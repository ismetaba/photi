# T06 Test Plan — Photo Endpoints + Job Queueing

## Acceptance Criteria
1. `POST /events/:id/photos` (multipart, multi-file). Owner only. For each file:
   - Store full bytes under `events/{eventId}/photos/{photoId}/full.{ext}`.
   - Generate 400px webp thumb stored at `events/{eventId}/photos/{photoId}/thumb.webp`.
   - Parse EXIF `takenAt`.
   - Insert Photo row with `status=processing`, empty matchedUserIds and faceVectors.
   - Enqueue Job (`process-photo`, `{photoId}`) with `status=queued`, `attempts=0`.
   Returns 201 with `{ created: PhotoListItem[] }`.
2. `GET /events/:id/photos?cursor=&limit=` (owner only). Returns `{ items: PhotoListItem[], nextCursor }` with signed URLs, status, matchCount, isFeatured. Default limit 50, max 200.
3. `DELETE /photos/:id` (owner only). Deletes DB row and removes both storage objects.
4. `PATCH /photos/:id` toggles `isFeatured` (owner). Broadcasts to foyer hub stub (we just record calls in a stub for tests).
5. Foyer hub stub interface in place for T09 to replace.

## Test Scenarios (Vitest)
- **Upload**: post 2 fixture JPEGs to a fresh event → 201, two Photo rows with status `processing`, two Jobs with type `process-photo` queued, storage has full + thumb keys.
- **Upload owner check**: outsider 403.
- **List**: returns items with `fullUrl` `/files/...`, `thumbUrl` `/files/...`, `matchCount` 0 initially, `status` `processing`.
- **List pagination**: with `limit=1` two uploads return 1 item plus `nextCursor`; following the cursor returns the second item and `nextCursor=null`.
- **Delete**: storage objects are removed; DB row gone.
- **Patch isFeatured**: toggles boolean; foyer hub stub `broadcast` called with `{ eventId, photoId, isFeatured }`.
- **Patch non-owner** 403.
