# T09 Test Plan — Foyer SSE Hub + Foyer Data

## Acceptance Criteria
1. `FoyerHub` exposes `subscribe(eventId, handler) → unsubscribe()` and `broadcast(evt)`.
2. `GET /events/:slug/foyer-data` public (no `x-user-id` required) returns `FoyerData`-shaped payload: event branding + featured photos (signed URLs) + counts (participants, ready photos, distributions).
3. `GET /events/:id/foyer-stream` returns `text/event-stream`, sends an initial `event: hello` message + keeps the connection open with periodic keepalives. Each subscriber receives subsequent broadcasts as `data:` lines. Closing the response unsubscribes.
4. Existing photo PATCH `isFeatured` flow continues to broadcast.

## Test Scenarios (Vitest)
- **FoyerHub.subscribe + broadcast**: subscribe → broadcast hits handler with payload; unsubscribe stops further deliveries; multiple subscribers all receive.
- **GET /foyer-data**: anonymous request (no `x-user-id`) returns 200; payload matches `FoyerData` schema; counts: 1 participant + 1 ready photo + 1 distribution after the happy-path setup.
- **/foyer-data 404 unknown slug**.
- **GET /foyer-stream**: returns 200 with `content-type: text/event-stream`. After registering a route subscriber, broadcasting should write a `data:` line. (We use a custom Node test that wires the foyerHub subscriber list directly to assert broadcast plumbing — keepalive timer disabled in tests.)
